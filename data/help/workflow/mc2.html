Microcode (MA2) Workflow

Applies to: MA2

Writing and Running Microprograms

1) Write your microprogram
Enter microcode statements in the Microcode pane, one cycle per line. Each statement specifies the control signals to assert in that cycle. Use Automatic Formatting to standardize spacing and layout.

Alternatively, you can interact with the CPU diagram's control signals to generate statements. After seting the control signals, click the Copy to Microcode button The corresponding statement is inserted at the end of the microcode pane.

To comment or uncomment a block of lines, select them and use Edit → Toggle line comment. Any partially selected line is also affected. Uncommented lines become commented, and commented lines become uncommented. This is useful for switching between unit tests: select both test blocks and toggle them in one operation.

2) Microassemble
Use Build → Microassemble or the toolbar button to check for errors without running the program. Errors appear inline in the Microcode pane in red.

3) Run
Use Build → Run or the toolbar button to microassemble and execute the program. On success, the parsed microcode appears in the Object Code pane and unit test results appear in the Test Results pane.

Debugging Microprograms

1) Start debugging
Use Debug → Start Debugging to microassemble, apply UnitPre conditions, and enter Debugger mode. The CPU diagram becomes active and shows the initial datapath state. Registers and memory are reloaded to their default values, except for any changes specified by UnitPres.

2) Step through execution
Use Debug → Single Step to execute one microcode cycle at a time. The current cycle is highlighted in the Microcode pane, and the CPU diagram updates to reflect the signals asserted by that cycle.

Related views: Memory Dump.

3) Stop debugging
Use Debug → Stop Debugging or the toolbar icon to end the session early.

Unit Tests

Prior to loading a microprogram and setting preconditions, the CPU's registers will be reset to their default states and memory will be zeroed. UnitPre sets specified register/memory values before execution begins. UnitPost defines expected values for registers and memory after the last cycle. Post conditions are evaluated at each step, but failures are only reported at the end of simulation and highlighted in red in the test pane.

Register values

Register width determines the valid hex range for UnitPre and UnitPost specifications:

  • IR is 3 bytes. IR=0xa is stored or tested as 0x00000a.
  • T1 is 1 byte. Values above 0xFF produce an out-of-range error.
  • All other registers are 2 bytes. PC=0x1 is stored or tested as 0x0001.
  • NZVC bits are 1 bit each. For example, N=1 is valid. N=0 is redundant in UnitPre since all values are cleared to zero first, but may be included for clarity.

Memory values

Memory is byte-addressable. Mem[0x1234]=0x5 sets or tests the single byte at address 0x1234 to 0x05.

Two adjacent bytes can be specified with a single two-byte value in the range 0x0100–0xffff:

Mem[0xabcd]=0x1122   ; equivalent to Mem[0xabcd]=0x11, Mem[0xabce]=0x22
Note: The one-byte and two-byte data bus variants of the MA2 CPU have different sets of control signals. The CPU pane shows the diagram for the variant used by the current project.
